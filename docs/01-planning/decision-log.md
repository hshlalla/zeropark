## ADR-001: 코어 스파인 (capability/provider/registry/router)

- **결정:** 제품 어휘(Capability) ↔ 엔진 인터페이스(Provider) ↔ Registry/Router 분리.
  기능 추가=`cap_*` 1개, 엔진 추가=클래스+등록 1줄. 중앙 분기문 금지.
- **상태:** 유효. 구현이 HTTP든 네이티브든 무관하게 재사용됨(피벗에도 살아남음).

## ADR-005: 네이티브 단일 프레임워크로 피벗 (ADR-002/003 대체)

- **맥락:** 초기엔 "엔진을 격리 서비스로 띄우고 HTTP 어댑터로 호출"을 택했으나,
  판매용 프레임워크에는 부적합. 고객이 5개 서비스를 운영해야 하고, 코드 소유·라이선스가 불명확.
- **결정:**
  1. 엔진을 **API로 호출하지 않는다.** OSS는 **설계 참고**만.
  2. 각 capability를 **퍼미시브 라이브러리로 네이티브 구현**(crawl=httpx+markdownify, slides=python-pptx 등).
  3. 전부 **하나의 리포/프레임워크**로 패키징. `external/`는 런타임에서 제외.
  4. `zeropark-adapters`(HTTP) 폐기 → `zeropark-engines`(네이티브)로 교체.
- **결과:**
  - 엔진을 import 하지 않으므로 FastAPI/Python 버전 충돌이 원천 소멸.
  - 단일 설치 아티팩트 → 클라이언트 구축이 쉬워지고 코드/IP를 소유.
  - 라이선스 위험을 코드 차용 정책으로 통제(Dify/SearXNG 비차용).
- **트레이드오프:** 5개 엔진 기능을 "유사하게" 점진 구현해야 함(범위 큼). crawl·slides부터 구현 완료.
- **세부:** [architecture/dependency-isolation.md](architecture/dependency-isolation.md)
