# 제품 요구사항 (Zeropark)

## 1. 제품 한 줄 설명

오픈소스를 최대한 활용해, 한 프롬프트로 리서치·슈퍼에이전트·슬라이드·시트·대시보드·웹
자동화·아티팩트를 생성하는 **Genspark형 AI 워크스페이스 프레임워크**. 타 기업에 구축
가능한 **솔루션(구축 서비스)** 으로 판매하는 것이 목표.

## 2. 문제 정의

이런 기능을 직접 풀스택으로 만들면 비용·시간이 크고, 개별 OSS는 강력하지만 의존성·라이선스·
운영이 제각각이다. 이를 **하나의 안정적 제품 API와 교체 가능한 어댑터 계층**으로 묶어, 기능을
계속 덧대도 스파게티가 되지 않고, 클라이언트별로 빠르게 구축·판매할 수 있어야 한다.

## 3. 목표

- 한 프롬프트로 여러 산출물(리서치/슬라이드/시트/대시보드/페이지)을 생성한다.
- 모든 무거운 기능은 검증된 OSS 엔진을 격리 통합해 재사용한다.
- 기능별로 객체화(어댑터/capability)되어 추가·교체가 국소 변경으로 끝난다.
- 한 배포 = 같은 코드 + 다른 설정. 클라이언트별 구축이 fork가 아니라 config.
- 라이선스/의존성 리스크를 코어에서 격리한다.

## 4. 주요 사용자

| 사용자 | 원하는 것 |
|---|---|
| 최종 사용자 | 프롬프트로 리서치/덱/시트/대시보드 생성, 소스·진행·다운로드 |
| 구축 대상 기업(고객사) | 자사 환경에 배포, 자사 모델/검색/도구로 구성, 화이트라벨 |
| 우리(프레임워크 제공자) | 빠른 구축, 엔진 교체, 라이선스 안전, 버전 재현성 |
| 개발자/AI 에이전트 | 기능을 안전하게 덧대기 위한 명확한 어댑터 계약 |

### 5.1 횡단 인프라 (capability 공통 선행 — 최우선)

여러 기능이 공유하므로 먼저 만든다: **LLM client 추상**(멀티프로바이더), **Tool/Skill 레지스트리**,
**Memory+Vector store**, **Sandbox(코드 실행)**, **Streaming(SSE)**, **Observability(토큰·비용·트레이스)**,
**MCP 클라이언트/서버**. 세부 갭·단계는 [capability-roadmap.md](capability-roadmap.md) 참고.

## 6. 제외 범위

- Genspark의 이름·UI·브랜드·비공개 프롬프트·전용 모델 복제.
- 라이선스가 허용하지 않는 OSS 브랜딩 제거/재판매.
- 출처 검증 없는 결과 제공(인용 환각은 제품 리스크).

## 7. 성공 기준

| 기준 | 목표 |
|---|---|
| 객체화 | 새 엔진 추가가 어댑터 1개 + 등록 1줄로 끝난다 |
| 격리 | 제품 코어가 어떤 엔진도 import 하지 않는다(버전 충돌 0) |
| 구성성 | 같은 코드가 설정만으로 다른 클라이언트 배포가 된다 |
| 라이선스 | 까다로운 엔진(Dify/SearXNG)이 교체형 어댑터 뒤에 격리된다 |
| 재현성 | 엔진 버전이 커밋 단위로 핀 고정된다 |

## 8. 의사결정 원칙

- 통합보다 격리. 결합보다 교체 가능성.
- 빠른 기능 추가보다 어댑터 계약의 안정성.
- 출처/라이선스/보안은 출시 전 필수 게이트.
